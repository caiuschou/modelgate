- 在**不改变「用户 API Key = 网关访问凭证」**这一核心定义的前提下，系统化描述 **Key 能力增强** 的产品范围、分期策略与**可落地的交互细节**。
- 为产品评审、设计与实现提供**单一事实来源**：验收标准、页面状态机、文案与错误码级预期。

- 产品经理 / 设计师：评审范围、走查流程与空态。
- 前后端工程师：接口形状、字段与边界行为。
- 安全与合规：存储、轮换与审计要求的对照清单。

## 二、当前基线（实现快照）

以下描述与仓库当前行为一致，作为增强设计的起点。

### 2.1 后端

- **认证：** `Authorization: Bearer <api_key>`；校验 `api_keys` 表且 `revoked = 0`。
- **用户侧接口：** `GET/POST /api/v1/me/api-keys`，`DELETE /api/v1/me/api-keys/{id}`（吊销）。
- **数据：** `api_keys` 存完整密钥明文（与 [api-key-management.md](api-key-management.md) 中「哈希或加密存储」的长期目标存在差距，增强路线中单独列出 **存储加固** 里程碑）。

### 2.2 前端（`/api-keys`）

- **列表：** 预览（掩码前缀）、创建时间、有效/已吊销、吊销操作。
- **创建：** 一键「新建密钥」→ 成功后在页面顶部 **一次性** 展示完整密钥，支持复制与「我已保存」。
- **缺失：** 命名、描述、最后使用时间、禁用（非吊销）、配额、模型/IP 策略、详情页、与日志的快捷跳转。

### 2.3 基线小结

当前能力满足 **多 Key 并存 + 吊销 + 单次展示明文**；增强重点在 **可运营性（命名/用量/策略）** 与 **安全运维（轮换体验、存储、管理审计）**。

## 三、增强愿景与能力框架

### 3.1 产品愿景（一句话）

让用户能为**每个应用或环境**分配一把 **可识别、可限制、可观测** 的密钥，并在泄露或人员变动时 **低风险轮换**，且与 **审计日志** 中的 `token_id`（或等价标识）天然对齐。

### 3.2 能力维度（用于分期与优先级）

| 维度 | 含义 | 典型能力 |
|------|------|----------|
| **身份与元数据** | Key 在人类界面可区分 | 名称、描述、标签、环境（dev/stage/prod） |
| **生命周期** | 创建 → 使用 → 停用 → 销毁 | 禁用/启用、吊销、过期时间、轮换向导 |
| **策略** | 网关在授权后 enforced 的规则 | 模型白名单、端点范围、IP 白名单、配额 |
| **可观测** | 用量与排障 | 最后使用时间、按 Key 聚合用量、异常告警入口 |
| **治理** | 谁做了什么 | 管理操作审计（创建/改策略/吊销的操作者与时间） |
| **安全基座** | 降低泄露面 | 存储哈希/加密、只显式一次、轮换流程 |

## 四、分期路线图

### 4.1 Phase A — 可运营（高优先级，强用户价值 / 中实现成本）

**目标：** 列表「看得懂」、创建「可备注」、运维「敢轮换」。

| 编号 | 能力 | 产品要点 |
|------|------|----------|
| A1 | **名称与可选描述** | 创建时必填名称（1–64 字符），描述可选（0–256）；列表展示名称为主标题，预览为辅。 |
| A2 | **最后使用时间** | 网关在有鉴权成功的请求上异步/同步更新 `last_used_at`（注意写入频率与性能，可采用节流或按分钟桶）。 |
| A3 | **轮换向导（产品流程）** | 非自动：引导「创建新 Key → 更新业务配置 → 验证流量 → 吊销旧 Key」；页面上提供**检查清单**与复制区。 |
| A4 | **从 Key 跳转到日志** | 列表行操作「查看相关日志」，预填 `token_id` 或等价过滤条件（与审计字段对齐）。 |
| A5 | **禁用（非吊销）** | 可逆停用：用于冻结排错；与吊销在文案与图标上明确区分。 |

### 4.2 Phase B — 策略与配额（中高优先级，依赖网关 enforcement）

**目标：** Key 不仅是身份，还是**策略载体**。

| 编号 | 能力 | 产品要点 |
|------|------|----------|
| B1 | **配额** | 按 Token 或按金额（与 [features.md](features.md#22-配额管理) 一致）；控制台展示剩余额度与重置周期。 |
| B2 | **模型白名单** | 空 = 不额外限制；非空 = 仅允许列表内模型 id。 |
| B3 | **IP 白名单** | CIDR 列表；空 = 不限制。需明确 **IPv4/IPv6** 与 `X-Forwarded-For` 信任策略（部署文档中约定）。 |

### 4.3 Phase C — 治理与安全加固（持续）

| 编号 | 能力 | 产品要点 |
|------|------|----------|
| C1 | **管理操作审计** | 记录：操作类型、目标 key id、操作者 user_id、时间、来源 IP（可选）。 |
| C2 | **存储加固** | 持久化仅存密钥指纹/哈希；校验时用常数时间比较；创建时唯一一次返回明文。 |
| C3 | **过期时间** | 可选 `expires_at`；过期等价于自动禁用，列表显示「已过期」。 |

## 六、交互设计 — 列表页（`/api-keys`）

### 6.1 页面结构（自上而下）

1. **页头：** 标题「API 密钥」、一句说明（Bearer 与 Token 计费的区分，保留现有文案并可略缩短）。
2. **主操作：** 主按钮「新建密钥」（Primary）；次要「了解轮换」（可选，打开侧抽屉或文档链接）。
3. **全局反馈区：** 创建成功的一次性密钥卡片（现有琥珀色强调区，保留并增强见 6.3）。
4. **数据区：** 表格（桌面）/ 卡片列表（移动端，见 6.6）。

### 6.2 表格列（Phase A 目标）

| 列 | 内容 | 说明 |
|----|------|------|
| 名称 | 主文案 + 可选环境标签 | 无名称的旧数据兼容：显示「未命名密钥」+ id 后缀 |
| 预览 | 掩码 | 与现有一致 |
| 最后使用 | 相对时间 + 绝对时间 tooltip | 从未使用：「从未使用」 |
| 状态 | 有效 / 已禁用 / 已吊销 / 已过期 | 色板与 [设计令牌](../design/interaction/design-tokens.md) 一致 |
| 操作 | 查看、禁用/启用、吊销、相关日志 | 已吊销行仅「查看」（只读历史） |

### 6.3 新建密钥 — 交互流程（推荐：对话框或侧抽屉）

**触发：** 点击「新建密钥」。

**步骤 1 — 元数据**

- 输入：**名称**（必填，实时校验长度与非法字符）。
- 输入：**描述**（可选，多行）。
- 可选：**环境** 单选（开发 / 测试 / 生产 / 未指定）— 仅展示与筛选用，可不进 v1。

**步骤 2 — 确认**

- 摘要展示将创建的内容；说明「完整密钥仅出现一次」。

**步骤 3 — 结果**

- 与当前页内卡片一致：完整密钥、`复制`、`我已保存`。
- **附加：** 勾选「打开轮换指引」时，展开简短检查清单（见 7.2）。

**取消规则：** 若在步骤 3 未点「我已保存」就关闭，需 **二次确认**「关闭后无法再查看完整密钥」。

### 6.4 行级操作与确认

| 操作 | 确认层级 | 主文案要点 |
|------|----------|------------|
| 禁用 | 轻确认（可 inline 或对话框） | 说明可逆、已有请求可能短暂失败 |
| 启用 | 无需或轻提示 | — |
| 吊销 | **强确认**（现有 `window.confirm` 可升级为模态） | 不可恢复、建议先轮换 |
| 相关日志 | 无 | 跳转日志中心并带 query |

### 6.5 空态与加载

- **空态：** 保留现有 EmptyState；主按钮与文案引导创建第一把 Key。
- **加载：** 表格骨架屏（与日志中心一致更佳）。
- **错误：** 行内错误 + 「重试」；401 统一走全局登出或登录提示。

### 6.6 移动端

- 表格改为 **卡片**：名称置顶，预览与状态次之，底部操作按钮组。
- 长密钥展示：**横向滚动** `pre` + 复制按钮固定 accessible name。

### 6.7 无障碍（A11y）

- 一次性密钥区域：`role="alert"` 或 `aria-live="polite"` 在创建成功时播报。
- 吊销确认：焦点陷阱在模态内，关闭后焦点回到触发按钮。
- 表格：行操作按钮带 `aria-label`（含 Key 名称避免重复「吊销」歧义）。

## 七、交互设计 — 轮换向导（流程型）

### 7.1 入口

- 列表工具栏「轮换指引」；详情页突出按钮「安全轮换」。

### 7.2 步骤（内容文案级）

1. **新建替代密钥** — 按钮跳转创建（或内嵌创建完成态）。
2. **在应用中更新** — 展示环境变量名示例（如 `MODELGATE_API_KEY`），提示勿提交到 Git。
3. **验证** — 建议用户发起一次测试请求或查看「相关日志」是否出现新 `token_id`。
4. **吊销旧密钥** — 选择要吊销的旧 Key（若多把），强确认。

### 7.3 非目标（v1）

- 不承诺「零失败窗口」的自动双写；产品定位为 **人工确认下的低风险流程**。

## 八、交互设计 — 详情页（`/api-keys/:id`）

### 8.1 区块划分

1. **概要：** 名称、预览、状态、创建时间、最后使用时间、过期时间（若有）。
2. **策略摘要（Phase B）：** 配额进度条、模型与 IP 列表或「未限制」。
3. **快捷操作：** 禁用、吊销、轮换、查看日志。
4. **活动（可选 Phase C）：** 近期管理操作时间线（谁在何时修改）。

### 8.2 只读与历史

- 已吊销 Key：全页只读，操作区仅「返回列表」「查看历史日志」。

## 九、API 与数据形状（草案）

以下为 **目标态 REST 形状**，实现时可微调路径，但字段语义建议保留。

### 9.1 列表 / 创建 / 更新

```
GET    /api/v1/me/api-keys
POST   /api/v1/me/api-keys
       Body: { "name": string, "description"?: string, "expires_at"?: number }
PATCH  /api/v1/me/api-keys/{id}
       Body: { "name"?: string, "description"?: string, "disabled"?: bool }
DELETE /api/v1/me/api-keys/{id}   # 吊销，与现有一致或语义化改为 POST /revoke
```

### 9.2 响应字段（列表项示例）

```json
{
  "id": 1,
  "name": "生产-支付助手",
  "preview": "sk-mg-v1-abc…xyz1",
  "created_at": 1710000000,
  "last_used_at": null,
  "status": "active",
  "revoked": false,
  "disabled": false,
  "expires_at": null
}
```

`status` 可由服务端派生：`revoked` > `expired` > `disabled` > `active`，减少前端分支错误。

### 9.3 数据表扩展（概念）

在现有 `api_keys` 上增量（名称仅为示意）：

- `name`, `description`
- `last_used_at`
- `disabled`（与 `revoked` 区分）
- `expires_at`（Phase C）
- 策略与配额：可拆子表 `api_key_policies` 或 JSON 列（需权衡查询与迁移）

## 十、与审计、计费的衔接

- **审计：** 请求日志已关联 `token_id`（或等价）时，控制台「相关日志」以该维度过滤；文档与 UI 使用同一术语。
- **计费 / 配额扣减：** 若按 Key 配额，扣减失败时返回 **402 或 429**（产品需在 [product/api.md](api.md) 与 [development/api.md](../development/api.md) 中统一）。

## 十一、非目标与开放问题

### 11.1 非目标（本文档 v1 不展开）

- 面向租户的 **子账户 RBAC**（多用户共享同一控制台账号下的 Key 权限差异）。
- 与上游 OpenRouter **Management API** 的密钥同步（ModelGate 用户 Key 与 OpenRouter 账户 Key 为不同概念）。

### 11.2 开放问题（需决策）

- **登录态与 API Key 是否拆分：** 当前登录返回的 token 与网关 Key 语义合并；若拆分，本页需增加「仅会话」与「长期 Key」的说明模块。
- **最后使用时间的精度：** 秒级 vs 分钟桶 vs 仅日期，影响性能与产品解释成本。

## 十二、验收与指标

### 12.1 Phase A 验收清单

- [ ] 创建时必须填写名称；列表以名称识别 Key。
- [ ] 旧数据无名称时有降级展示。
- [ ] 最后使用时间在真实流量下单调更新（允许节流）。
- [ ] 禁用与吊销在 UI 与 API 上语义区分清晰。
- [ ] 「相关日志」跳转带正确过滤参数。
- [ ] 轮换向导文案与步骤在 E2E 中可被覆盖（可选）。

### 12.2 成功指标（建议）

- 用户为第二把 Key 命名率 > 80%（相对创建次数）。
- 吊销前用户平均持有 Key 数量 > 1（说明轮换路径被使用）。
- Key 相关支持问题（无法区分环境）类工单下降（需基线）。
