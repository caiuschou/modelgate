# ModelGate API 密钥（Key）管理 — 产品说明

**版本:** 1.1  
**更新日期:** 2026年4月  
**文档类型:** 产品规格（用户侧访问凭证）

---

## 一、术语与定位

### 1.1 两类「密钥」

| 概念 | 说明 | 谁持有 |
|------|------|--------|
| **用户 API Key（访问凭证）** | 调用 ModelGate 网关时的身份凭证，请求头 `Authorization: Bearer <key>`（勿与模型用量 **Token** 混淆） | 开发者 / 业务应用 |
| **上游供应商 Key** | ModelGate 代表用户访问 OpenAI 兼容等服务时使用的真实密钥 | 运维在服务端配置（如 `UPSTREAM_API_KEY`），**不**通过控制台发给终端用户 |

本文档聚焦 **用户 API Key** 的**产品能力**：创建、分发、轮换、吊销、权限与配额、与审计的关系。上游渠道密钥属于 **渠道管理** 产品域（见 [产品概述](overview.md) 与 [功能详解 — 渠道管理](features.md#一渠道管理功能)）。

### 1.2 产品目标

- **隔离：** 业务只接触 ModelGate 颁发的 Key，降低上游主 Key 扩散面。  
- **可追溯：** 每次请求可关联到 `user_id` / `token_id`，支撑审计与成本分摊（见 [请求审计日志](audit-log.md)）。  
- **可管控：** 支持多把 Key（按项目/环境拆分）、吊销与（规划中的）配额、IP/模型范围等策略。

## 二、密钥生命周期（目标体验）

### 2.1 创建

**用户故事：**

```
作为 开发者或管理员
我想要 在控制台或通过管理流程创建新的 API Key
以便于 为新应用或环境分配独立凭证，且不影响其他 Key
```

**产品要求：**

- 支持 **命名 / 描述**（便于在列表中识别，如「生产-支付助手」）。  
- 生成后 **完整密钥仅展示一次**，引导用户立即复制；后续列表中仅显示掩码前缀（如 `sk-or-v1-****`）。  
- 可选：**过期时间**、（规划中）**配额上限**、**可调用接口类型**、**模型白名单**、**IP 白名单**。

**验收要点：** 创建流程宜控制在 **3 步内**（命名与策略 → 确认 → 展示密钥），见 [功能详解 — 2.1 创建 API 密钥](features.md#21-创建-api-密钥)。

### 2.2 查看与列表

- 列表展示：名称、掩码 Key、创建时间、最后使用时间（若可采集）、状态（启用/禁用/过期）。  
- **不可**在详情中再次查看完整密钥；若遗失，应走 **轮换**（新建 + 停用旧 Key）。

### 2.3 轮换

**用户故事：**

```
作为 安全负责人
我想要 在不停服的前提下轮换泄露风险的 Key
以便于 先签发新 Key、业务切换后再吊销旧 Key
```

**产品要求：** 支持同一用户下 **多 Key 并存**；吊销旧 Key 后，该 Key 的后续请求返回 **401**，审计中仍可按 `token_id` 检索历史记录。

### 2.4 吊销与禁用

- **吊销（Revoke）：** 永久失效，不可恢复；用于泄露或员工离职场景。  
- **禁用（Disable）：** 临时停用以配合排错或冻结账号，可再次启用（规划能力，以最终实现为准）。

### 2.5 与登录态的关系（控制台）

控制台登录成功后返回的 **`token` 字段** 在语义上即为 **可用于调用网关 API 与审计查询 API 的同一类 Bearer Key**（产品上一致：减少用户记忆多套凭证的成本）。若未来拆分为「会话 JWT + 独立 API Key」，需在交互上明确区分；当前开源原型采用 **登录即返回 API Key** 模式，见 [开发 API — 登录](../development/api.md)。

## 三、权限模型（产品规划）

### 3.1 调用 OpenAI 兼容 API

默认：持有有效用户 Key 即可调用网关已开放的端点（如 `POST /v1/chat/completions`）。  
扩展：按 Key 限制 **可访问模型**、**端点类型**（Chat / Embeddings / Images 等），矩阵见 [功能详解 — 2.3 权限控制](features.md#23-权限控制)。

### 3.2 管理类能力

- **普通 Key：** 仅调用推理类接口与（按数据范围过滤后的）**本人**审计查询。  
- **管理员：** 额外访问控制台中的用户管理、渠道配置等（与「管理员 API 密钥」或 RBAC 角色绑定，见 [用户控制台 — 3.3](user-console.md#33-api-密钥管理) 与功能详解权限表）。

### 3.3 网络与来源限制（规划）

- **IP 白名单：** 未配置则不限制；配置后仅允许列表内来源。  
- **域名限制：** 针对浏览器端场景，校验 `Origin` / `Referer`（见 [功能详解](features.md#23-权限控制)）。

## 四、配额与用量（产品规划）

按 Key 或按用户聚合：**Token 数、金额、时段** 等维度；扣减规则与预警阈值见 [功能详解 — 2.2 配额管理](features.md#22-配额管理)。  
用量展示可与 **日志中心**、**统计分析** 联动，按 `token_id` 过滤（见 [审计日志产品说明](audit-log.md)）。

---

## 五、典型场景

1. **新项目接入：** 管理员创建 Key → 配置（可选）IP/模型 → 交付给开发 → 在日志中心按 Key 维度看调用。  
2. **密钥泄露：** 立即吊销旧 Key → 创建新 Key → 业务侧更新环境变量 → 确认审计中无新异常来源。  
3. **多环境隔离：** 开发/测试/生产各一把 Key，吊销生产 Key 时不影响测试环境。  

与控制台整体故事一致的部分见 [用户控制台 — 四、用户场景](user-console.md#四用户场景)。

## 六、安全与合规要求（产品层）

- 密钥 **禁止** 在 URL Query 中作为主要方式（易进访问日志）；推荐 **Authorization Bearer**。  
- 服务端存储：**哈希或加密存储**完整密钥、仅生成时明文回传（具体算法以安全设计为准）。  
- 操作审计：谁在何时创建/吊销了哪把 Key（规划中的**管理操作审计**，与**请求审计**区分）。  
- 合规：密钥操作与访问日志保留策略需可配置，与 [审计日志](audit-log.md) 保留周期对齐。

## 八、与当前开源原型的差异（实现状态）

| 能力 | 产品目标（本文） | 当前仓库（摘要） |
|------|------------------|------------------|
| 控制台密钥页（列表/创建/吊销） | ✅ 规划 | `/api-keys` 为占位页 |
| 登录返回 Key | ✅ | ✅ `POST /api/v1/auth/login` |
| 注册时自动创建 Key | ✅ | ✅ 随用户写入 `api_keys` |

| 多 Key 并存 | ✅ | ✅ 数据层可支持多条 `api_keys` |
| 吊销 / 配额 / IP 白名单 | ✅ 规划 | ❌ 未形成完整产品流程 |
